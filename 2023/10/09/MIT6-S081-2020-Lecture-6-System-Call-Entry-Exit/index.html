参考资料：

trap机制简单介绍

当计算机遇到以下三种情况时，会从用户态转换到内核态调用相应的handler：

  1. 用户态使用系统调用
  2. 用户态遇到硬件中断
  3. 用户态遇到异常，如除0，page fault

MIT教材将这种用户态和内核态的切换称为trap机制。主要流程如下：

  1. 通过某个入口进入trampoline page执行其中的跳转代码
  2. 执行uservec.S保存用户态下的寄存器至trapframe page,并且从trapframe中恢复内核态下的一些状态到寄存器中
  3. 执行usertrap.c,该函数根据scause记录的产生trap的原因调用不同的handler进行处理
  4. 处理
  5. 执行usertrapret.c,将内核状态保存到trapframe用与再次进入内核态，设置一些寄存器让硬件帮助跳回到用户态
  6. 执行userret.S,恢复用户态的状态到寄存器
  7. 执行sret回到用户态

gdb调试trap机制

根据mit 6.s081 lecture6的介绍，对xv6使用系统调用时的用户态-内核态以及内核态-用户态转变机制的跟踪

主要流程：

  • sh调用write系统调用
  • 在users.S中可以看到调用了ecall进入内核
  • ecall会跳入stvec寄存器设置的地址执行，具体为trampoline.S
  • 然后分别执行uservec,usertrap.c,syscal.c,sys_write.c,usertrapret.c,userret
  • 回到ecall继续执行原来的程序

  • 执行continue到ecall断点位置

  • 需要明确ecall会做的事情，ecall是risc-v提供的指令，主要做了三件事情

    • 将用户态转换为内核态
    • 将当前程序的pc保存到sepc寄存器
    • 把stvec存储到地址读取到pc中，并跳转执行

  • 观察当前状态

    在运行qemu的窗口执行ctrl-a c,然后输入info mem观察页表状态,可以推测当前是用户态，其中2000这一页是stack guard page,3fffffe000是trapframe,3ffffff000是trampoline page

  • 观察当前pc和sepc和stvec

    可以看到当前pc指向了dec这条指令，也就是ecall，且处于用户态。sepc指向了df8,查看sh.asm,发现是close调用的ret指令;stvec指向3ffffff000指向了trampoline page,根据xv6设置和lecture 6讲座内容可以知道，trampoline page中存放了trampoline.S汇编代码，且stvec会在上次内核态切换到用户态时设置为trampoline page的地址，（问：第一次怎么设置的？答：系统在启动时会先进入内核态，然后切换到用户态，所以每次进入用户态都是从内核态切换进入到。）

  • 输入info register,观察寄存器信息，可以看到a0-a2存储write的参数，a7存储write调用在内核中对应的值

  • step进入ecall

    由于我们知道ecall会执行trampoline page中的代码,我们在trampoline page打上断点,然后continue

  • 观察ecall执行后的状态,打印pc寄存器，可以看到当前已经出去trampoline page并且这个page是禁止用户访问的，所以此时已经在内核态了，但是我们打印进程页表，发现仍然是用户页表，这是因为satp仍然指向着原来的页表，因此需要在trampoline.S中更新为内核页表。（问：为什么切换到内核态后仍然可以执行trampoline.S的代码？答：因为trampoline.S在内核态和用户态使用同样的映射位置）

  • 观察执行ecall后的pc和sepc和stvec

    可以看到ecall已经帮我们把原来pc的值保存到sepc中了，stvec的值没有改变，但是ecall跳转了stvec值的位置继续执行。

  • 对trampoline.S汇编指令逐行调试，根据顺序，trampoline.S会先执行uservec代码。

  • 需要找到一块区域来存放用户状态下32个寄存器的值，这块区域需要满足能在用户态和内核态都能找到，xv6使用高地址trapframe来实现。

    1. 当前我们在内核态但使用的是用户态的pagetable，和我们的调试结果一样。

    2. sscratch保存了进程的trapframe

  • 加载trapframe地址

    使用了指令csrrw a0, sscratch, a0,效果是交换a0和sscratch两个寄存器的值,我们执行step info,然后打印两个寄存器可以看到

  • 恢复上一次从内核态转换到用户态时到内核态上下文，主要包括4个

    1. kernel_sp内核态下的栈低指针
    2. kernel_hartid内核态下的cpuid
    3. kernel_trap user_trap()函数的地址
    4. kernel_satp 内核态下pagetable的地址，然后赋值给satp寄存器

  • 跳转执行usertrap.c,文件位于kernel/trap.c

  • usertrap.c可以被分为这几块

    1. 判断是否是用户态转换到内核态，如果不是，panic

    2. 将kernelvec函数的地址赋值给stvec寄存器，因为我们现在在内核状态，如果再次需要trap机制时，不需要trampoline.S的机制，内核状态下的trap机制和用户态不同，此处先不讨论

    3. 将sepc寄存器的值放到trapframe中，lecture 6上有同学问为什么不放到tramppline.S中一起保存。Robert Tappan Morries教授回答：可以，没有明显的理由要放在哪儿一边。

    4. 根据scause寄存器的值决定采取的动作，分为三种：系统调用，异常，硬件中断

    5. 判断程序是否应该被kill

    6. 调用usertrapret函数

  • gdb启动布局

  • 观察stvec寄存器

    可以看到此时stvec已经不在指向trampoline.S,我们打印该地址的指令，可以看到是kernelvec

  • 系统调用引发的trap下usertrap做的事

    可以看到主要有3件事

    1. 将trapframe中的epc+4，是因为trapfram中的epc将作为切换回用户态时的地址，可以看到dec的下一条指令地址为df0,相差4bytes

    2. 打开中断，由此我们可以推断ecall帮我们关闭了中断？（不确定）

    3. 调用syscall函数，该函数根据a7寄存器调用sys_write,具体不在展开，此处只调试trap

  • 在运行完syscall后会进入usertrapret函数，该函数主要做了

    1. 关闭中断，因为此时需要切换状态，不能被中断打断

    2. 设置stvec的值,此处有w_stvec(TRAMPOLINE+uservec-trampoline)

      • w_stvec是内联汇编，设置stvec寄存器的值

      • TRAMPOLINE是一个宏，位于kernel/memlaout.h

      • MAXVA和PGSIZ位于kernel/riscv.h,分别表示最高虚拟地址0x4000000000和页大小4096

      • 因此可以得到TRAMPOLINE就是trampoline.S的首地址，因为trampoline page位于最高地址

      • uservec和trampoline的值可以打印出来，发现都指向uservec

      • 在运行这句后，打印stvec的值，发现就是TRAMPOLINE.S,不太清楚为什么要写成这句，或者说为什么要加上(uservec-trampoline)?

    3. 将当前内核状态保存到trapframe中，用于下次进入内核态时恢复

    4. 设置一些寄存器给riscv帮助恢复到用户态并打开中断

    5. 设置sepc值,可以看到sepc寄存器已经设置为了需要跳回的地址

    6. 将当前用户态的pagetable地址赋值给stap变量，并在后续作为参数传递给userret用来恢复pagetable

      此处使用MAKE_SATP宏对pagetabl做了一些处理

      因为xv6对虚拟页表地址格式如下，后12位表示偏移，此处将pagetable右移12位方便后续操作。

    7. 跳转到fn,fn对应trampoline.S中到userret

      根据上面的经验，可以看到fn对应的trampoline.S中userret的地址0x3ffffff090

  • userret主要做了几件事

    1. 恢复用户态pagetable
    2. 恢复寄存器
    3. 调用sret，让硬件帮助我们回到用户态

  • 因为userret带着参数，我们可以先看下当前寄存器的状态,输入指令

    可以看到a0和a1保存着我们传进来的参数trapframe和用户态的page table地址。

    但是，sys_write的返回值去哪儿了？，一般来说，返回值会放到a0寄存器给sh使用，但是此时a0寄存器已经被覆盖了。

    morries 教授回答是在sscratch寄存器。

    后面发现不对，此时的sscratch应该是一看是uservec中的a0寄存器的值（write到标准错误2）

    但是是什么时候把返回值放到sscratch寄存器的？？应该不是放到sscratch寄存器，而是在syscall时放到了trapframe中。

    我们在usertrap阶段打印trapframe也能看到112偏移位置已经被修改成了1
